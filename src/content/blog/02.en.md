## Introduction

Single Responsibility Principle is usually summarized as “a class should have only one reason to change”, but in React this is not always obvious.

The problem appears when a component starts doing too much: it renders UI, manages state, validates data, calls services, and also handles business logic. At first it feels practical, but over time it becomes fragile.

SRP is not about splitting code just for the sake of it, but about understanding responsibilities and reducing the cost of change.

## The core idea

SRP does not mean that a component has to be small, but that its responsibilities should be clear.

A component can have many lines and still be well designed, if all of them respond to the same reason for change. The problem starts when we mix different decisions in the same place.

For example:

- the UI changes
- validation changes
- submit logic changes
- a business rule changes

If all of that lives in the same file, any change forces us to touch the same piece.

The useful question is not “is this component too big?”, but:

**How many real reasons does it have to change?**

## Practical example

A common case:

```jsx
const AuthForm = () => {
  const [values, setValues] = useState({
    user: "",
    password: "",
  });

  const handleSubmit = (e) => {
    e.preventDefault();

    if (!values.user || !values.password) {
      alert("Required fields");
      return;
    }

    sendAuth(values);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={values.user}
        onChange={(e) =>
          setValues({
            ...values,
            user: e.target.value,
          })
        }
      />

      <input
        type="password"
        value={values.password}
        onChange={(e) =>
          setValues({
            ...values,
            password: e.target.value,
          })
        }
      />

      <button>Login</button>
    </form>
  );
};
```


At first glance, it looks fine. This component “handles” authentication. But in reality, the form is managing state, validations, and submit logic.

This means that a UI update, a validation change, or a business rule update can all require changes in the same file.

A more modular approach could be:

- one component for rendering or presentation
- another one for validations
- another one to handle the side effect of form submission

This separation makes reading, testing, and reusing the components easier.

## Slightly serious closing thoughts

The principle of single responsibility is not about the number of lines in a file. It is a way to identify and reduce, when it makes sense, the number of reasons a component may need to change.

If a component includes UI, validations, side effects, and/or business rules, it is usually a sign that it may be doing too much.

The goal is not to compulsively create more files, but to make them clear to read, easy to use (and reuse), and safe to change.

### Questions to apply SRP in React
- What decisions is this component making?
- How many different reasons would it have to change?
- Is this component mixing domain logic with presentation logic?
- If validation changes, should this file need to change?
- If the UI changes, should the state logic change too?
- If a component starts mixing UI, validation, side effects, and business rules, it usually means responsibilities are drifting together.
