## Introducción

Single Responsibility Principle suele resumirse como “una clase debería tener una sola razón para cambiar”, pero en React eso no siempre resulta evidente.

El problema aparece cuando un componente empieza a hacer demasiado: renderiza UI, maneja estado, valida datos, llama servicios y además decide lógica de negocio. Al principio parece práctico, pero con el tiempo se vuelve frágil.

SRP no se trata de dividir por dividir, sino de entender responsabilidades y reducir el costo de cambio.

## La idea central

SRP no implica que un componente tenga que ser pequeño, sino que sus responsabilidades sean claras.

Un componente puede tener muchas líneas y seguir estando bien diseñado, si todas responden al mismo motivo de cambio. El problema empieza cuando mezclamos decisiones distintas en el mismo lugar.

Por ejemplo:

- cambia la UI
- cambia la validación
- cambia la lógica de envío
- cambia una regla de negocio

Si todo eso vive en el mismo archivo, cualquier cambio obliga a tocar la misma pieza.

La pregunta útil no es “¿este componente es grande?”, sino:

**¿Cuántas razones reales tiene para cambiar?**

## Ejemplo práctico

Un caso común:

```jsx
const AuthForm = () => {
  const [values, setValues] = useState({
    user: "",
    password: "",
  });

  const handleSubmit = (e) => {
    e.preventDefault();

    if (!values.user || !values.password) {
      alert("Campos obligatorios");
      return;
    }

    sendAuth(values);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={values.user}
        onChange={(e) =>
          setValues({
            ...values,
            user: e.target.value,
          })
        }
      />

      <input
        type="password"
        value={values.password}
        onChange={(e) =>
          setValues({
            ...values,
            password: e.target.value,
          })
        }
      />

      <button>Login</button>
    </form>
  );
};

```

A primera vista luce bien. Este componente “se ocupa” de la autenticación. Pero en realidad, el formulario está manejando estado, validaciones y lógica de envío.

Esto implica que una actualización de la UI, de validaciones o de reglas de negocio puede llevar a modificar el mismo archivo.

Una posibilidad más modular podría ser:

- un componente de renderizado o presentación
- otro para validaciones
- otro que maneje el efecto secundario del envío del formulario

Esta separación facilita la lectura, el testeo y el uso de los componentes.

## Reflexiones sesudas para cerrar

El principio de responsabilidad única no refiere al número de líneas de un archivo. Es un criterio para identificar y reducir, en lo posible y cuando conviene, la cantidad de motivos de cambio para un componente.

Si un componente incluye UI, validaciones, efectos secundarios y/o reglas de negocio, suele ser una señal de que está abarcando demasiado.

El objetivo no es multiplicar compulsivamente los archivos, sino que los mismos sean claros de leer, fáciles de usar (y reutilizar) y seguros de cambiar.


### Preguntas guía para aplicar SRP en React:


- ¿Qué decisiones está tomando este componente?
- ¿Cuántos motivos distintos tendría para cambiar?
- ¿Este componente mezcla lógica de dominio con lógica de presentación?
- Si cambia la validación, ¿debería cambiar este archivo?
- Si cambia la UI, ¿debería cambiar la lógica de estado?
